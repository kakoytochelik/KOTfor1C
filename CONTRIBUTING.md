# Как вносить изменения

Спасибо за интерес к проекту.

`KOT for 1C` развивается как практическое расширение VS Code для команд, которые пишут и сопровождают YAML-сценарии 1С, используют Vanessa Automation, `KOT Infobase Manager` и `KOT Form Explorer`. Вклады в код, документацию, интерфейс, локализацию и исправления ошибок приветствуются.

Обсуждения в issue и PR можно оформлять на русском или английском языке.

## С чего начать

- Для мелких исправлений документации, текстов или локальных исправлений можно сразу открывать PR.
- Для крупных изменений поведения, новых команд, новых настроек или заметных изменений интерфейса лучше сначала описать идею в issue или в черновике PR, чтобы согласовать объем работы.
- Перед началом полезно прочитать:
  - [`README.md`](./README.md)
  - [`documentation/QUICK_START.md`](./documentation/QUICK_START.md)
  - [`documentation/SETUP.md`](./documentation/SETUP.md)
  - [`documentation/DEVELOPMENT.md`](./documentation/DEVELOPMENT.md)

## Локальный запуск

1. Установите зависимости:

```bash
npm install
```

2. Соберите расширение:

```bash
npm run compile
```

3. Откройте проект в VS Code и запустите `Extension Development Host` через `F5`.

4. Для итеративной разработки можно использовать:

```bash
npm run watch
```

## Что важно учитывать при изменениях

- Проект написан на TypeScript и собирается через `esbuild`.
- В репозитории сейчас нет настроенного набора автоматических тестов: `npm test` намеренно сообщает `No automated tests configured`.
- Если вы меняете команды, настройки, сообщения, заголовки веб-панелей или другой пользовательский текст, обновляйте локализацию:
  - [`package.nls.json`](./package.nls.json)
  - [`package.nls.ru.json`](./package.nls.ru.json)
  - [`l10n/bundle.l10n.json`](./l10n/bundle.l10n.json)
  - [`l10n/bundle.l10n.ru.json`](./l10n/bundle.l10n.ru.json)
- Если меняется поведение расширения, обновляйте документацию в [`documentation/`](./documentation/) и, при необходимости, [`README.md`](./README.md).
- Старайтесь держать PR узким по объему: одна задача, одна заметная цель, минимум побочных рефакторингов.

## Минимальная проверка перед PR

Перед отправкой изменений выполните:

```bash
npm run compile
```

Затем вручную проверьте только те области, которых касается изменение.

- Для редактора сценариев: автодополнение, подсказки при наведении, диагностика, быстрые исправления и действия при сохранении.
- Для `Test Manager`: чекбоксы, фильтрацию сборки, запуск Vanessa, статусы и обновление лога в реальном времени.
- Для `Parameters Manager`: вкладки, импорт/экспорт, поиск, при сборке результат итогового json.
- Для `Infobase Manager`: открытие веб-панели и целевые операции с ИБ.
- Для `Form Explorer`: запуск, чтение снимка состояния формы, режимы `manual/auto` и состояние панели.

Если меняется основной поток выполнения или архитектура, ориентируйтесь на чеклист из [`documentation/DEVELOPMENT.md`](./documentation/DEVELOPMENT.md).

## Что указать в PR

- Кратко опишите проблему и решение.
- Укажите, что именно проверяли вручную.
- Для изменений интерфейса приложите скриншоты/видео/GIF, если это помогает оценить результат.
- Если изменение затрагивает совместимость, явно опишите риск регрессии и ожидаемое поведение после обновления.
- Не коммитьте артефакты сборки, локальные логи, `.vsix`, `node_modules`, содержимое `out/` и прочие временные файлы выполнения.

## Что обычно не стоит делать

- Не смешивать большой рефакторинг с функциональным изменением без необходимости.
- Не менять формат существующих YAML-сценариев без понятной причины и описания миграционного эффекта.
- Не добавлять новые пользовательские строки только в один язык интерфейса.
- Не оставлять изменение без обновления документации, если поведение команды, настройки или интерфейс реально изменились.
