# Блок: Parameters Manager

## Задача блока

Дать единое место для управления параметрами сборки и запуска Vanessa Automation.

Открытие: в панели Менеджера тестов иконка "`{ }`" или команда `KOT - Open Build Scenario Parameters Manager` (`KOT - Открыть Менеджер параметров Сборки Сценариев`).

## Вкладки менеджера

## 1) СППР

Используется для параметров, которые обрабатывает `СборкаТекстовСценариев`.

Результат:

- формируется `yaml_parameters.json`;
- эти параметры участвуют в сборке `feature/json`;
- после сборки KOT использует их как основу для built JSON и соседнего `*.combined.json` артефакта Vanessa.

### Минимальный набор для сборки

KOT ориентируется на фактическое поведение `СборкаТекстовСценариев` из СППР.

Жестко обязательны в KOT:

| Ключ | Для чего нужен |
|---|---|
| `ScenarioFolder` | Каталог, откуда обработка читает YAML-сценарии. KOT синхронизирует из него внутренний корень сканирования. По умолчанию: `tests/`. |
| `FeatureFolder` | Каталог, куда записываются результаты компиляции. Без него сборка падает с ошибкой `Не указан каталог для результатов компиляции`. |
| `VanessaFolder` | Каталог Vanessa Automation. По умолчанию KOT берет его из `Run Vanessa: Vanessa Epf Path`. |
| `LaunchDBFolder` / `TestClientDBPath` / `InfobasePath` / `TestClientDB` | Каноничный launch-путь к базе для JSON Vanessa. В UI это одна обязательная строка с выбором alias-ключа. |

Условно обязательны:

| Ключ | Когда нужен |
|---|---|
| `ModelDBSettings` | Формально обязателен только при `AuthCompile=true`, но на практике почти всегда нужен, если вы хотите, чтобы СППР автоматически подставлял логин/пароль `TestClient` по `ПрофильПользователя` и умел разрешать `DT` эталонной базы. По умолчанию: `.vscode/kot-runtime/bases.yaml`. |

Часто используемые, но необязательные:

| Ключ | Для чего нужен |
|---|---|
| `ProcessFolder` | Если процессы лежат не там же, где сценарии. |
| `CompileFile` | Явный путь к `compile.txt`, если не хотите использовать путь по умолчанию в `FeatureFolder`. |
| `CreateJson`, `CreateJUnit` | Управление генерацией вспомогательных файлов результатов. |
| `LogFile`, `ResultFile` | Пути к лог-файлу и файлу кода результата обработки. |

Нюанс обработки СППР:

- ключи параметров нормализуются;
- обработка игнорирует регистр, пробелы, `-` и `_`;
- поэтому `FeatureFolder`, `feature_folder` и `feature-folder` для нее эквивалентны.

Нюанс KOT:

- `FeatureFolder` управляет именно выходными `feature/json` артефактами СППР;
- настройка расширения `kotTestToolkit.runtime.directory` используется как отдельная служебная runtime-папка KOT для `yaml_parameters.json`, логов и временных файлов сборки; по умолчанию это `.vscode/kot-runtime`;
- это разные роли, и их не стоит смешивать концептуально, даже если физически пути могут быть рядом.

## 2) Дополнительные параметры Vanessa

Используется для VAParams-ключей, которые не покрываются обработкой СППР полностью.

Результат:

- параметры мержатся в `combined JSON` рядом с built JSON;
- исходный built JSON от СППР при этом не переписывается;
- для каждой строки можно включить `Override existing value`, если нужно переопределить одноименный ключ из СППР.

## 3) GlobalVars

Используется для пользовательских глобальных переменных, которые мержатся в итоговый VAParams при запуске.

Фактически эти значения тоже попадают в `combined JSON`, который KOT использует для built-in запуска Vanessa.

Эти переменные также доступны в редакторе через подсказки `$$GlobalVar$$`.

## Профили и autosave

- Менеджер параметров хранит несколько профилей наборов параметров.
- Активный профиль выбирается из шапки панели или из меню кнопки `Build tests` в Менеджере тестов.
- Все изменения сохраняются автоматически.
- В шапке показывается `Updated at`, чтобы было видно, когда последний раз сохранилось текущее состояние.

## Операции в менеджере

- Редактирование ключей, значений и флагов `Override existing value`.
- Поиск по имени параметра на каждой вкладке.
- Для вкладки `СППР`: предпросмотр `yaml_parameters.json`, импортировать JSON, сбросить fixed-значения к default-набору.
- Для вкладки `Дополнительные параметры`: сохранить runtime JSON, импортировать VAParams/JSON, очистить секцию.
- Для вкладки `GlobalVars`: сохранить JSON, импортировать JSON, очистить секцию.

Что дополнительно помогает делать KOT:

- во вкладке `СППР` незаполненные обязательные поля подсвечиваются прямо в таблице;
- строки `ScenarioFolder`, `FeatureFolder`, `VanessaFolder`, `ModelDBSettings`, `AuthCompile` и launch-база закреплены вверху и не удаляются;
- launch-база настраивается как одна фиксированная строка, но с выбором alias-ключа `LaunchDBFolder` / `TestClientDBPath` / `InfobasePath` / `TestClientDB`;
- `AuthCompile` остается отдельным режимом строгой валидации, а не источником логинов/паролей;
- если `ModelDBSettings` пустой, KOT явно предупреждает, что СППР не сможет автоматически разрешать логины/пароли `TestClient` по профилю;
- если в импортированном JSON был `TestFolder`, в UI он нормализуется в строку `ScenarioFolder`, потому что для СППР они эквивалентны;
- остальные известные ключи СППР можно добавлять через searchable picker с кратким описанием;
- `ScenarioFilter` и `ExceptScenario` не добавляются вручную, потому что KOT формирует их автоматически по выбору чекбоксов в Test Manager;
- для модифицированных обработок можно добавлять и полностью произвольные custom-строки;
- при предпросмотре `yaml_parameters.json` KOT валидирует минимальный набор параметров;
- при автосохранении KOT не показывает отдельный apply-диалог, но все равно помечает обязательные незаполненные поля и сохраняет текущее draft-состояние профиля;
- при автосохранении KOT синхронизирует внутренний корень сканирования из `ScenarioFolder` и запускает пересканирование списка тестов.

## Приоритеты и merge-логика

- Значения из СППР считаются базовыми.
- Дополнительные параметры позволяют дополнять `combined JSON` и точечно переопределять одноименные ключи.
- `GlobalVars` тоже умеют точечно перезаписывать существующее значение, если у строки включен `Override existing value`.
- По умолчанию при конфликте одноименного ключа значение из СППР выигрывает.

## Где это особенно полезно

- Нужно добавить параметр VA, который не поддерживается СППР «из коробки».
- Нужны проектные/локальные runtime-ключи без изменения стандартных шаблонов сборки.
- Нужно хранить и переиспользовать наборы GlobalVars.
