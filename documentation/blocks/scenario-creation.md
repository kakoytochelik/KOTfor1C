# Блок: Создание сценариев

## Задача блока

Создавать новые главные и вложенные сценарии с согласованной шапкой YAML для `СборкаТекстовСценариев`, Test Manager и служебных функций расширения.

## Что входит в блок

### Команды

- `KOT - Создать вложенный сценарий`
- `KOT - Создать главный сценарий`
- `KOT - Управление функциями системы`
- `KOT - Управление эталонными базами`

### Что создает расширение

- Для вложенного сценария:
  - папку `<Код>/`;
  - файл `scen.yaml`;
  - папку `files/`.
- Для главного сценария:
  - папку `<Имя>/`;
  - файл `scen.yaml` на основе шаблона `main.yaml`;
  - файл `test/<Имя>.yaml` на основе шаблона `test.yaml`;
  - папку `files/`.

### Поток создания

1. Расширение запрашивает обычные пользовательские данные:
   - для вложенного сценария: `Имя`, `Код`, родительскую папку;
   - для главного сценария: `Имя`, вкладку Test Manager, порядок на вкладке, значение чекбокса по умолчанию, родительскую папку.
2. Затем выбирается `ФункцияСистемы`:
   - если в каталоге настроена одна функция, она берется автоматически;
   - если функций несколько, показывается `QuickPick`.
3. После этого расширение спрашивает, как заполнить остальные служебные поля:
   - взять значения из `kotTestToolkit.newScenarioDefaults`;
   - или пройти по каждому полю вручную, но с предзаполнением значениями по умолчанию.
4. Для ручного режима по очереди запрашиваются:
   - `Проект`;
   - `РазрешеноИспользоватьВДругихФункциях`;
   - `ПрофильПользователя`;
   - `УровеньОтчета1`;
   - `УровеньОтчета2`.
5. Для главного сценария дополнительно выбираются данные `test.yaml`:
   - эталонная база из текущего `ModelDBSettings` (`bases.yaml`);
   - профиль пользователя из выбранной эталонной базы.
6. Если `bases.yaml` еще не заполнен, KOT не блокирует создание сценария:
   - можно создать тест и заполнить `ЭталоннаяБазаИмя`, `ИдентификаторБазы`, `ПрофильПользователя` позже;
   - для этого в `test.yaml` есть отдельные действия по карандашу.
7. После записи файлов расширение открывает созданный сценарий и обновляет Test Manager.

## Каталог `ФункцияСистемы`

### Как устроен

- Каталог хранится в настройке `kotTestToolkit.newScenarioDefaults.systemFunctions`.
- Каждая запись содержит пару:
  - `name` — значение `ДанныеСценария.ФункцияСистемы`;
  - `uid` — значение `ДанныеСценария.UIDФункцияСистемы`.
- Первая запись в списке считается значением по умолчанию.

### Как редактируется

- Через команду `KOT - Управление функциями системы`.
- В менеджере можно:
  - просмотреть текущий список;
  - добавить новую запись;
  - изменить имя;
  - изменить UID;
  - удалить запись;
  - сделать запись значением по умолчанию.

### Важное поведение

- Для новой функции системы UID генерируется автоматически.
- В YAML всегда записывается согласованная пара `ФункцияСистемы` + `UIDФункцияСистемы`.
- Это нужно, потому что `СборкаТекстовСценариев` использует `UIDФункцияСистемы` и `РазрешеноИспользоватьВДругихФункциях` для ограничения вызовов сценариев между функциями системы.

## Поля шапки сценария

Ниже перечислены основные поля, которые расширение подставляет при создании сценария.

| Поле | Для чего нужно | Кем используется |
|---|---|---|
| `ТипФайла` | Отличает сценарий от файла настройки теста | `СборкаТекстовСценариев`, расширение |
| `ДанныеСценария.UID` | Основной идентификатор сценария | `СборкаТекстовСценариев`, Test Manager |
| `ДанныеСценария.Имя` | Человекочитаемое имя сценария | `СборкаТекстовСценариев`, расширение |
| `ДанныеСценария.Код` | Код сценария для тегов, JSON и диагностики | `СборкаТекстовСценариев`, расширение |
| `ДанныеСценария.Проект` | Проектный контекст сценария | в первую очередь `СборкаТекстовСценариев` |
| `ДанныеСценария.ФункцияСистемы` | Имя функции системы, к которой относится сценарий | `СборкаТекстовСценариев`, расширение |
| `ДанныеСценария.UIDФункцияСистемы` | GUID той же функции системы | `СборкаТекстовСценариев` |
| `ДанныеСценария.РазрешеноИспользоватьВДругихФункциях` | Разрешает или запрещает вызов из другой функции системы | `СборкаТекстовСценариев` |
| `ДанныеСценария.ПрофильПользователя` | Профиль TestClient по умолчанию | `СборкаТекстовСценариев`, запуск тестов |
| `ДанныеСценария.УровеньОтчета1` | Источник `#report.feature` в итоговом `feature` | `СборкаТекстовСценариев` |
| `ДанныеСценария.УровеньОтчета2` | Источник `#report.story` в итоговом `feature` | `СборкаТекстовСценариев` |
| `ПараметрыСценария` | Объявление параметров сценария | `СборкаТекстовСценариев`, расширение |
| `ВложенныеСценарии` | Связи с вызываемыми сценариями | `СборкаТекстовСценариев`, расширение |
| `ТекстСценария` | Основной текст Gherkin-сценария | `СборкаТекстовСценариев`, расширение |
| `KOTМетаданные.Описание` | Пользовательское описание сценария | расширение |
| `KOTМетаданные.PhaseSwitcher.Tab` | Вкладка сценария в Test Manager | расширение |
| `KOTМетаданные.PhaseSwitcher.OrderOnTab` | Порядок внутри вкладки | расширение |
| `KOTМетаданные.PhaseSwitcher.Default` | Значение чекбокса в Test Manager по умолчанию | расширение |

### Что важно помнить

- Поля `ДанныеСценария.*` в основном нужны для совместимости с `СборкаТекстовСценариев`.
- Поля `KOTМетаданные.*` нужны самому расширению и не участвуют в BDD-сборке.
- Для большинства проектов разумно держать служебные значения в defaults и редко менять их вручную.

## Что записывается в `test.yaml`

`test.yaml` создается только для главного сценария и хранит настройки запуска теста.

| Поле | Назначение |
|---|---|
| `ТипФайла: "НастройкаТеста"` | Помечает файл как тестовую настройку |
| `ДанныеТеста.UID` | Отдельный UID самой настройки теста |
| `ДанныеТеста.UIDСценария` | Ссылка на `ДанныеСценария.UID` главного сценария |
| `ДанныеТеста.Имя`, `Код`, `СценарийНаименование` | Отображаемое имя и код теста |
| `ДанныеТеста.ПрофильПользователя` | Профиль запуска; обычно выбирается из профилей текущей эталонной базы |
| `ДанныеТеста.ЭталоннаяБазаИмя` | Отображаемое имя эталонной базы из `bases.yaml` |
| `ДанныеТеста.ИдентификаторБазы` | Каноничный идентификатор эталонной базы из `bases.yaml`; по нему СППР ищет `DT`, логин и пароль |
| `ДанныеТеста.ПараметрыЗапуска`, `ПараметрыТеста` | Дополнительные параметры запуска и настройки теста |

## Редактирование `test.yaml`

В редакторе `test/*.yaml` доступны действия по карандашу:

- для `ЭталоннаяБазаИмя` / `ИдентификаторБазы`:
  - открыть picker эталонных баз;
  - синхронно обновить оба поля;
- для `ПрофильПользователя`:
  - выбрать профиль из текущей эталонной базы;
- для `Код`, `UIDСценария`, `Имя`, `СценарийНаименование`:
  - открыть меню `Change linked scenario`;
  - переименовать связанный сценарий;
  - синхронизировать поля из соседнего `scen.yaml`;
  - перелинковать `test.yaml` на другой главный сценарий.

## Быстрый переход из `scen.yaml`

- В шапке главного `scen.yaml` доступно быстрое действие `Open test settings`.
- Оно открывает связанный `test.yaml`, не требуя искать его через дерево проекта или контекстное меню Test Manager.
- Подробнее о каталоге эталонных баз: [etalon-bases.md](./etalon-bases.md)

## Настройки блока

| Настройка | Назначение |
|---|---|
| `kotTestToolkit.newScenarioDefaults.project` | Значение по умолчанию для `ДанныеСценария.Проект` |
| `kotTestToolkit.newScenarioDefaults.systemFunctions` | Каталог доступных функций системы |
| `kotTestToolkit.newScenarioDefaults.allowCrossFunctionUsage` | Значение по умолчанию для `РазрешеноИспользоватьВДругихФункциях` |
| `kotTestToolkit.newScenarioDefaults.userProfile` | Профиль пользователя по умолчанию для `scen.yaml` и `test.yaml` |
| `kotTestToolkit.newScenarioDefaults.reportLevel1` | Значение по умолчанию для `УровеньОтчета1` |
| `kotTestToolkit.newScenarioDefaults.mainReportLevel2` | Значение по умолчанию для `УровеньОтчета2` главного сценария |
| `kotTestToolkit.newScenarioDefaults.nestedReportLevel2` | Значение по умолчанию для `УровеньОтчета2` вложенного сценария |
| `kotTestToolkit.editor.newScenarioLanguage` | Язык `#language` в новом сценарии |
| `ScenarioFolder` из Менеджера параметров | Базовая папка для выбора местоположения нового сценария |
| `ModelDBSettings` из Менеджера параметров | Источник `bases.yaml`, из которого KOT предлагает эталонные базы и профили для `test.yaml` |
| `kotTestToolkit.assembleScript.showDriveFeatures` | Добавлять ли в главный сценарий стартовый 1C:Drive-блок |

## Практические советы

- Если проект использует один и тот же набор служебных значений, держите их в `newScenarioDefaults` и не дублируйте вручную в каждом YAML.
- Если поменялась функция системы, изменяйте ее через менеджер функций системы, а не через несвязанные строки в настройках.
- Если хотите, чтобы новые `test.yaml` сразу создавались с корректной эталонной базой и профилем, сначала заполните `ModelDBSettings` / `bases.yaml`.
- Переименование главного сценария синхронно обновляет его имя, `Код`, папку сценария и основной файл в `test/`.
- Наведением на ключи шапки в YAML можно посмотреть краткое описание поля прямо в редакторе, в том числе для `test.yaml`.
